## 背景與問題

使用者有兩種身分，現在的 UI 把兩者的功能混在同一條流程裡，越來越雜：

- **玩家／主持本機遊戲**：生成題組 → 在本 app 的墨水遊戲畫面玩。現有主流程。
- **出題者（本計畫的主角）**：生成題組後，**不在本 app 玩**，而是把每題回答的注音登錄給遊戲平台的 BOT。BOT 指令格式（新版，具名參數）：

```
/ghostink clue 題目id:5 題組:4 選項:1 注音:ㄐㄧㄢˉㄔㄠˇ
/ghostink clue 題目id:5 題組:3 選項:1 注音:ㄐㄧㄣˉ
```

出題者相關的既有零件：`groupPaste.ts`（貼上「第 N 組」文字 → (題組, 選項) 標籤，存在 `settings.groupTags`）、`hostCommands.ts`（**舊版位置參數**格式 `/ghostink clue 1 3 ㄐㄧㄢˉㄘㄞˋ`，顯示在玩家結果畫面的折疊區 — 這正是要搬走的雜亂來源）。

3. **單題重生＝累積候選挑一個**：每按一次「再生一個」新增候選，全部並列、各附指令與複製鈕，點選「採用」標記。
4. **題目id**：出題者設定頁輸入（記住上次值），指令頁上也能改，改了所有指令即時更新。

## 資料模型與核心契約

### `hostCommands.ts` 改寫（Phase 1）

```ts
export interface ClueParams {
  prefix: string;      // 'ghostink' | 'phantomink' | 自訂（不含斜線與 clue）
  questionId: string;  // 題目id，保持字串（BOT 那邊未必是數字）
  group: number;       // 題組
  option: number;      // 選項（= groupPaste 的 index，1-based）
  zhuyin: string;      // toBopomofoCells(reply).join('')
}

export function buildClueCommand(p: ClueParams): string {
  return `/${p.prefix} clue 題目id:${p.questionId} 題組:${p.group} 選項:${p.option} 注音:${p.zhuyin}`;
}
```

舊的 `buildClueCommands`／`CLUE_CMD_PREFIX` 刪除，`hostCommands.test.ts` 重寫（見 Phase 1 測試清單）。

### 指令頁的畫面狀態（Phase 2/3，放 `hostMode.ts` 模組內）

```ts
interface HostCard {
  question: string;
  tag: { group: number; option: number } | null; // 來自 groupTags 匹配；null = 未匹配，需手動補
  candidates: string[];  // 回答候選；[0] 是本次生成的原始回答
  adopted: number;       // 採用中的候選 index，預設 0
}
```

- 匹配方式沿用 `normalizeQuestion`（見 groupPaste.ts）。
- settings 新欄位：`mode?: 'host' | 'player'`、`hostQuestionId?: string`、`cmdPrefix?: string`。**記得 bump `schemaVersion`**（settings.ts 已有版本重置機制）。

### 單題重生（Phase 3，`generator.ts` 加公開方法）

```ts
/** 為單一題重生一個回答候選。走 CODE 硬擋，不合格自動重試，最多 3 次。 */
async regenerateReply(
  answer: string,
  question: string,
  opts: { avoid: string[]; rejected: string[] },
): Promise<string>
```

- prompt 要求：回答 ≤6 中文字、全中文、不含謎底任何字、不得與 `avoid`（其他題目前採用的回答）重複、不要與 `rejected`（本題已有候選）相同或同義、結尾句號。走 `this.jsonChat`（自然套用 json_object＋fallback 鏈）。
- 回傳前跑與 `generate()` 相同的硬擋（`replyCharCount ≤ 6`、非空、不含謎底字、不在 avoid 中）；3 次都不合格 → throw 帶原因的 Error（UI 顯示，不加入候選）。

## UI 流程

### 開場模式選擇（新）

- app 啟動：`settings.mode` 已存在 → 直接進該模式；否則顯示選擇畫面。
- 兩個模式的第一個畫面（設定頁）頂部都有小連結「⇄ 切換模式」→ 清掉 `settings.mode` 回到選擇畫面（其他設定保留）。

### 出題者設定頁（精簡）

保留區塊：API Key（含申請說明）、Backend/Model、謎底模式（AI 出題／自己輸入謎底）、**貼上題組**（此模式的主要輸入）、題目id 欄、指令前綴選單。
**不顯示**：題庫勾選器、N/M 數量欄、老天有眼等玩家相關設定。

生成參數（重用 `PhantomInkGenerator.generate`，不改其邏輯）：
- 貼上解析結果 `matchToBank` 後：`pickedBankQuestions` = matched 的題庫原文、`customQuestions` = unmatched 原文；`numQuestions` = 解析出的總題數（M == forced 已被允許，見 commit ebbdfa3）；`numCandidates` 不傳。
- 沒貼題組就按生成 → 擋下並提示「出題者模式請先貼上題組」。

### 指令頁（生成完成後進入；取代玩家的遊戲畫面）

```
題目id:[ 5 ]  前綴:[ghostink ▾]           [📋 複製全部（21）]
謎底：鋼琴（AI 生成）                      [🔄 重新生成整組]

┌ Q（題組 4・選項 1）它的重量和什麼相仿？────────────
│ ● 候選1（採用中）：一台冰箱。   注音：ㄧˊㄊㄞˊㄅㄧㄥˉㄒㄧㄤˉ
│   /ghostink clue 題目id:5 題組:4 選項:1 注音:ㄧˊㄊㄞˊㄅㄧㄥˉㄒㄧㄤˉ   [📋]
│ ○ 候選2：兩袋白米。   注音：⋯⋯
│   /ghostink clue 題目id:5 題組:4 選項:1 注音:⋯⋯   [📋]
│ [🔄 再生一個候選]
└──────────────────────────────
┌ Q（⚠️ 無題組編號）自訂題目文字？ ─ 題組:[  ] 選項:[  ]
│ ●候選1：⋯⋯（編號填好前不顯示指令，顯示「請補題組/選項」）
└──────────────────────────────
```

行為規格：
- 每張卡：Q（含題組/選項小欄位，**可編輯**——這是「解析錯誤可自己替換」的需求）；候選列表（radio 採用）；每個候選一行單行指令＋單行複製鈕；「🔄 再生一個候選」按鈕。
- 題目id、前綴、任何 題組/選項 欄位變動 → 所有指令文字**即時重算**。
- 「複製全部」：每卡取採用中候選的指令，按（題組, 選項）排序、`\n` 連接；有卡片缺編號或注音為空時，該卡跳過並在按鈕旁顯示「（略過 N 題：缺編號）」。
- 單行複製用 `navigator.clipboard.writeText`，成功後按鈕短暫顯示 ✅（參考 solver 的複製回饋寫法）。
- 回答無漢字（注音為空）→ 卡片顯示「⚠️ 此回答無法轉注音」，不產指令。
- 重生中：該卡按鈕 disabled＋顯示進度（onEvent 事件字串也導到這裡）；失敗顯示原因，不新增候選。

### R1（裁決項）Phase 3 候選累積 — 等使用者決定

未裁決前不要動。→ 使用者裁決結果：＿＿＿＿

### R2（HIGH）模式切換鎖死
`main.ts` `main()`：只有 `!settings` 才顯示模式選擇；三處「⇄ 切換模式」都是清 `mode` 後 reload → 有 settings 但 mode undefined 會直接進玩家頁。**玩家按切換 = reload 回玩家頁自己（死連結）**；host 按切換 = 單向掉進玩家頁。
修法：`if (!settings || !settings.mode)` 顯示選擇畫面（callback 內以 `loadSettings() ?? 預設` 為基底，保留既有 apiKey 等設定）。
驗收：host↔選擇畫面↔player 三向都能到；既有設定（apiKey）切換後仍在。

### R3（HIGH）手動補題組/選項被重繪清空
`hostMode.ts`：題組/選項 input 的 `input` 事件觸發 `renderFullPage`/`updateAllCards` 全量 innerHTML 重繪，但輸入值從未寫回 `cards[idx].tag`，input 重繪後恢復成原始值（未匹配題＝空白）→ 打一個字就失焦＋值被吃掉，「解析錯誤可自己替換」實際上不可用；且指令文字與輸入框顯示會不一致。
修法：input 事件第一步把 `parseInt(value)` 寫回 `cards[idx].tag = {group, option}`（無效值存 0），重繪一律從 `cards` 讀值（刪掉 `getEffectiveGroup/Option` 查 DOM 的做法）；或改為不重繪 input、只更新該卡指令行與頂部計數。
驗收：未匹配卡片依序輸入 題組=9、選項=2，輸入過程不失焦、值不消失，指令行即時出現且複製全部包含該題。

### R4（HIGH）`regenerateReply` 缺 CODE 硬擋（重演 FL-1）
`generator.ts:594`：只在 prompt 裡要求規則，回傳前沒有任何檢查，`avoid`/`rejected` 參數整個沒實作。重生可能拿到超過六字、含謎底字、空字串、或與其他題重複的回答，UI 直接採用——正是 `docs/FAILURE-LOG.md` FL-1 與 QUESTION-QUALITY 原則 5 明令用 CODE 硬擋的情境。
修法（照本文件原契約）：簽名改 `regenerateReply(answer, question, opts: { avoid: string[]; rejected: string[] })`；prompt 帶入 avoid/rejected；回傳前檢查 `replyCharCount ≤ 6`、非空、不含謎底任何字、不在 avoid；不合格自動重試（總計 ≤3 次），仍不合格 throw 帶原因的 Error。呼叫端（hostMode.ts）傳 `avoid = 其他卡片現用回答`、`rejected = [被換掉的回答]`，catch 時把 `err.message` 顯示在卡片上（現在是 `catch {}` 吞掉、只閃 ❌）。
驗收：generator.test.ts 補：超長回覆自動重試、3 次不合格 throw、avoid/rejected 出現在 prompt 內（檢查 messages）。

### R5（HIGH）`updateAllCards` 重繪後「🔄 再生一個」全部失效
`hostMode.ts` `updateAllCards()` 重掛了複製與編輯 listener，漏掛 `.pi-host-regenerate-one` → 在題目id/前綴打過任何一個字之後，所有再生按鈕死掉。
修法：與 R3 一併把事件改成**事件委派**（在 `#pi-host-cards` 容器上掛一次 click/input，按 class 分派），從根本消除「重繪漏掛」這類 bug。
驗收：改題目id 後按任一「再生一個」仍有反應。

### R6（MEDIUM）指令頁「⇄ 切換模式」重繪後失效
切換連結的 listener 掛在 `renderHostCommands` 末尾（只執行一次），但連結本身在 `renderFullPage` 的 innerHTML 裡——任何觸發 `renderFullPage` 的編輯後連結變死的。修法：掛進 `renderFullPage` 內，或一併納入 R5 的事件委派。

### R7（MEDIUM）「🔄 重新生成整組」回設定頁後貼上區是空的
`renderHostSetup` 的 textarea 沒從 settings 還原 → 使用者得重貼全部題目。修法：從 `settings.groupTags` 重建貼上文字（`第 N 組` 標題＋逐行題目，group 遞增分段）填入 textarea 並觸發一次 `updateParseStatus()`。
