# 題目品質規範（出題／驗題的唯一標準）

> **地位**：改 `web/src/generator/prompts.ts`（出題、驗題、模擬 prompt）或 `generator.ts` 的品質檢查前必讀。
> 需求方 **raccoon** 實際試玩後定下的原則**凌駕一般直覺**——AI 直覺上「應該擋」的事，很多在這個遊戲裡是錯的。

## 遊戲背景（為什麼規則長這樣）

「靈媒遊戲」（Phantom Ink 變體）：主持人有一個謎底，玩家靠一組「問題＋回答」猜謎底；回答以**注音逐格揭露**（開墨水），所以流程**不可回推**——玩家看到的是殘缺注音，不是完整回答。

## raccoon 五原則（2026-07 試玩後修正，優先級最高）

1. **不要難度遞增／遞減。** 實際遊玩本來就有些線索關鍵、有些沒幫助，不需要刻意排序。任何「前幾題要難、後幾題要簡單」的檢查或 prompt 措辭都是錯的。
2. **「過於明確指向謎底」不用擋。** 因為注音是逐格揭露、不可回推，回答再明顯也不會提前破解。「太好猜」不是缺點，**是優點**。
3. **可解性是最高原則。** 有限的問答要真的能讓玩家答出來。七題湊出答案本身就很難，AI 不需要另外刁難。出題與驗題都以「猜得出來」為核心。
4. **「資訊重複且聚焦同一邏輯」要擋。** 這是 raccoon 唯一明確要求保留並加強的檢查：多題實質上在問／答同一件事（不只一字不差才算），會浪費題目名額、降低可解性。
5. **回答必須短（≤6 個中文字）。** 開墨水遊戲，回答太長直接破壞遊戲。**必須用 CODE 硬擋**，不能只靠 AI 驗題（AI 曾放行 14 字回答，見 FAILURE-LOG FL-1）。

**歷史教訓（Why）**：早期 AI 常「跑偏」——難度檢查亂擋好題、放行超長回答、human 模式產生非題庫的怪問題。五原則是實際試玩後的修正方向，不是理論推導。

## 其他硬性約束（與五原則並行）

- 謎底必須是**具體名詞**；臺灣慣用詞彙（大陸用語要換掉，`checkAnswerLocale` + `zhconv` 把關）。
- 回答不能直接出現謎底的任何中文字（間接指向可以，直接寫出不行）。
- 回答必須全中文、可轉注音、單句短語、結尾句號。
- 問題必須出自題庫（`QUESTION_BANK`）或使用者自訂；AI 不可自創問題文字。
- 三階段流程：出題（designer）→ 驗題（reviewer）→ 模擬玩家（simulator，預設關閉）。詳見 `memory/user_design_intent.md`。
  ⚠️ 該記憶檔與 README 仍寫著早期的「七題由難到易」設計——**已被 raccoon 原則 1 取代**，以本文件為準。

## 執行方式對照表（每條規則在哪裡把關）

| 規則 | 把關方式 | 位置 |
|---|---|---|
| 回答 ≤6 中文字 | **CODE 硬擋** | `generator.ts` `replyCharCount` + generate() 的 bad-set 修題迴圈 |
| 回答重複 | **CODE 硬擋** | 同上（重複 reply 檢查） |
| 回答洩漏謎底字 | **CODE 硬擋** | 同上（逐字比對） |
| 空回答 | **CODE 硬擋** | 同上 |
| 資訊重複／同一邏輯（原則 4） | **AI 驗題** | `prompts.ts` REVIEWER_SYSTEM_PROMPT |
| 可解性（原則 3） | **AI 驗題**＋（可選）模擬 | REVIEWER prompt；`simulatePlayer`（skipSimulation 預設 true） |
| 臺灣用語 | AI 檢查＋字轉換 | `checkAnswerLocale`、`zhconv.ts` |
| 問題必須出自題庫 | CODE 標記＋fix prompt 約束 | `generator.ts` isCustom 標記、`fixQuestions` 的題庫抽樣 |
| 難度排序 | **禁止檢查**（原則 1） | `prompts.test.ts` 有回歸測試擋「由難到易」字樣重新混入 |

**分工原則**：可程式判定的規則一律 CODE 硬擋；AI 驗題只負責語意判斷（重複邏輯、可解性）。這個分工不可反過來。

## 改動流程

1. 對照五原則逐條檢查你的改動有沒有違反（特別是「不要加難度要求」「不要擋明確線索」）。
2. 改 prompt 後跑 `cd web && npm test`（`prompts.test.ts` 有品質規則的回歸測試，必要時補測試）。
3. 手動生成一組題目確認輸出格式沒壞（見 `docs/ENGINEERING.md` 的驗收流程）。
4. 涉及規則本身的新共識（例如 raccoon 給了新修正），更新本文件並在 FAILURE-LOG 記錄背景。
